UVM Phase Architecture

UVM phases form a structured, synchronized execution framework that ensures every component in the testbench is built, connected, exercised, and torn down in the correct order. Phases are either top-down (parent before child), bottom-up (child before parent), or concurrent across all components.

⬇ Top-Down Phases

Parent executes before its children. Used when parent must configure children before they initialize themselves.

🔵 build_phase — parent runs first
↓
🔵 child build_phase runs next
Top-Down

⬆ Bottom-Up Phases

Children execute before their parent. Used when parent needs all children to be ready before completing its own phase.

🟢 child connect_phase runs first
↑
🟢 parent connect_phase runs next
Bottom-Up

⚡ Concurrent Run Phase

All components execute their run_phase simultaneously as parallel threads. Synchronization via events, semaphores, and the objection mechanism.

🟢 driver.run_phase ║ monitor.run_phase
🟢 agent.run_phase ║ scoreboard.run_phase
All Concurrent

🗓 Phase Execution Order

1.  build_phase — Top-down  function
2.  connect_phase — Bottom-up  function
3.  end_of_elaboration_phase — Bottom-up  function
4.  start_of_simulation_phase — Bottom-up  function
5.  run_phase — Concurrent  task  KEY PHASE
   5a. pre_reset → reset → post_reset → pre_configure → configure → post_configure
   5b. pre_main → main_phase → post_main
   5c. pre_shutdown → shutdown_phase → post_shutdown
6.  extract_phase — Bottom-up  function
7.  check_phase — Bottom-up  function
8.  report_phase — Bottom-up  function
9.  final_phase — Top-down  function
💡 Key Rule Phases 1–4 are the elaboration domain. Phase 5 is the runtime domain. Phases 6–9 are the cleanup domain. No phase in a later domain can start until ALL components have completed the current phase. This synchronization is automatic.

🧪 Test Class Phases

build_phase — MANDATORY. Test creates the environment here. Also where factory overrides are applied.

run_phase — MANDATORY. The test raises objections, starts virtual sequences, and drops objections. This IS the test body.

start_of_simulation — Optional. Print testbench hierarchy for debugging via this.print().

report_phase — Optional. Print test-level pass/fail summary.

⚠ No connect_phase in Test Tests don't connect TLM ports. They create the env which handles all connections.

🌍 Environment Phases

build_phase — MANDATORY. Creates all sub-components (agents, scoreboard, coverage, virtual sequencer).

connect_phase — MANDATORY. Wires TLM ports between agents and scoreboard/coverage. Sets vif handles in virtual sequencer.

end_of_elaboration — Optional. Used only if virtual sequencer fetches handles from config_db (alternative pattern).

No run_phase — Environments typically don't drive stimulus themselves. They delegate to agents.

🤖 Agent Phases

build_phase — MANDATORY. Creates driver, sequencer, monitor. Also retrieves the virtual interface from config_db.

connect_phase — MANDATORY. Connects driver's seq_item_port to sequencer's seq_item_export. Passes vif to driver/monitor.

No run_phase — Agent delegates runtime to driver and monitor.

🚗 Driver Phases

run_phase — MANDATORY. The forever loop: get sequence item → drive it → signal item_done. This is the driver's entire purpose.

build_phase — Optional. Only if driver needs to create sub-objects or retrieve config from config_db.

No connect_phase — Driver doesn't connect ports; the agent does that for it.

👁 Monitor Phases

run_phase — MANDATORY. Samples the interface signals and broadcasts observed transactions via analysis port.

build_phase — Optional. Only to retrieve vif (some patterns do this in agent).

No connect_phase — Analysis port connections are made by the agent or env, not the monitor itself.

📋 Sequencer Phases

No build_phase — uvm_sequencer's parent class already handles the built-in TLM port creation in its own build_phase. You don't need to override it unless adding config_db lookups.

No run_phase — The sequencer's arbitration loop is built into the base class. It just mediates between sequences and the driver automatically.

end_of_elaboration — Only if sequencer is a virtual sequencer fetching sub-sequencer handles from config_db.

🎯 Scoreboard Phases

build_phase — Optional but common. If scoreboard creates internal queues or retrieves config.

check_phase — MANDATORY. Compare actual vs expected, report pass/fail. At this point all stimulus has completed.

report_phase — Optional. Print detailed comparison summary.

No run_phase — Scoreboards react to incoming analysis port transactions via write() callbacks, not run_phase.

📊 Coverage Phases

report_phase — MANDATORY. Call $get_coverage() and print functional coverage results.

Coverage collectors work via write() callbacks from analysis ports — purely reactive, no run_phase needed.

Factory Overrides

UVM's factory allows you to substitute one class for another at runtime — without changing the original testbench code. There are two types: Type Override (global) and Instance Override (targeted).

🔄 Type Override

Replaces every instance of a class with the override class. Applied globally across the entire testbench.

Where: In test's build_phase() — BEFORE creating the env.

test_directed.sv SystemVerilog
class test_directed extends test_base; `uvm_component_utils(test_directed) function void build_phase(uvm_phase phase); // TYPE OVERRIDE: EVERY ahb_seq replaced globally ahb_seq::type_id::set_type_override( ahb_directed_seq::get_type() ); // MUST call super AFTER overrides, BEFORE env creation super.build_phase(phase); endfunction endclass

🎯 Instance Override

Replaces only a specific instance identified by its hierarchical path. All other instances stay as original.

test_mixed.sv SystemVerilog
class test_mixed extends test_base; `uvm_component_utils(test_mixed) function void build_phase(uvm_phase phase); // INSTANCE OVERRIDE: only THIS specific agent ahb_agent::type_id::set_inst_override( ahb_agent_err::get_type(), "uvm_test_top.env.ahb_agnt" // exact path ); // ETH agent is untouched — still original class super.build_phase(phase); endfunction endclass

📌 What Goes in build_phase — Complete List

💡 build_phase is the ONLY phase where components are created This is enforced by UVM's phasing: sub-components cannot exist before their parent's build_phase runs (top-down ordering guarantees this).
① Create child components ▼

All sub-components must be created using the factory's type_id::create() method — never with new() directly (or you lose factory override capability).

env.sv — build_phase
function void build_phase(uvm_phase phase); super.build_phase(phase); // ALWAYS call super first! // Create child components via factory ahb_agnt = ahb_agent::type_id::create("ahb_agnt", this); eth_agnt = eth_agent::type_id::create("eth_agnt", this); scoreboard = my_scoreboard::type_id::create("scoreboard", this); v_sqr = vsequencer::type_id::create("v_sqr", this); endfunction
② Retrieve virtual interface from config_db ▼

Virtual interfaces are retrieved in build_phase because you need them before run_phase. The agent retrieves the vif and stores it locally. It is then passed to driver/monitor in connect_phase.

my_agent.sv — build_phase
class my_agent extends uvm_agent; virtual my_if vif; // stored here my_driver drv; my_sequencer sqr; my_monitor mon; function void build_phase(uvm_phase phase); super.build_phase(phase); // ① Create sub-components drv = my_driver::type_id::create("drv", this); sqr = my_sequencer::type_id::create("sqr", this); mon = my_monitor::type_id::create("mon", this); // ② Retrieve virtual interface from config_db if(!uvm_config_db#(virtual my_if)::get( this, "", "my_vif", vif)) `uvm_fatal("NOVIF", "VIF not found in config_db!") endfunction function void connect_phase(uvm_phase phase); // ③ Pass retrieved vif to driver and monitor drv.seq_item_port.connect(sqr.seq_item_export); drv.vif = vif; // assign in connect_phase mon.vif = vif; // assign in connect_phase endfunction endclass
③ Apply factory overrides (in test only) ▼

Factory overrides MUST be registered before the components they affect are created. In the test's build_phase, register overrides BEFORE calling super.build_phase() which triggers the env creation chain.

test.sv — Override order matters!
function void build_phase(uvm_phase phase); // ✅ CORRECT: override registered BEFORE super my_driver::type_id::set_type_override( err_driver::get_type() ); super.build_phase(phase); // env created here endfunction // ❌ WRONG: override after super has no effect! function void build_phase_WRONG(uvm_phase phase); super.build_phase(phase); // env already created! my_driver::type_id::set_type_override( err_driver::get_type() // TOO LATE! ); endfunction
④ Retrieve configuration objects from config_db ▼
my_agent.sv — config retrieval
function void build_phase(uvm_phase phase); super.build_phase(phase); my_cfg cfg; // Retrieve configuration object if(!uvm_config_db#(my_cfg)::get( this, "", "agent_cfg", cfg)) `uvm_fatal("NOCFG", "No config found") // Use config to conditionally create components if(cfg.is_active == UVM_ACTIVE) begin drv = my_driver::type_id::create("drv", this); sqr = my_sequencer::type_id::create("sqr", this); end mon = my_monitor::type_id::create("mon", this); endfunction

Virtual Interface Handling

The canonical UVM pattern: retrieve the vif in build_phase, connect it to child components in connect_phase.

🔌 Why This Two-Phase VIF Pattern?

Why in build_phase?

The vif must be retrieved before connect_phase because connect_phase needs it to pass to child components. Since build_phase is the initialization phase, it's the correct place to fetch resources from config_db.

Additionally, if vif retrieval fails with `uvm_fatal, the simulation stops cleanly before any connections are attempted — better error reporting.

Why assign to children in connect_phase?

Children (driver, monitor) only exist after their own build_phase completes. Since build_phase is top-down, the agent's build creates the driver/monitor, then the driver/monitor's build runs. Only after all builds complete does connect_phase begin — so it's the earliest safe point to assign the vif to children.

Purpose

🏭 UVM Factory

How it works, why it matters, and the critical difference between new() and ::type_id::create().

The factory is a global run-time look-up table. Register your class → create it via the factory → any test can swap it for a compatible subtype without touching a single source file. That is the entire idea.

The core idea

In normal SystemVerilog, drv = new("drv", this) hard-codes exactly what gets built. The factory breaks that coupling: you register a type under a name, and when you ask for that name later the factory can return a different but compatible type — chosen at test time, not compile time.

Three things that make it a "factory"

1
Registration — every class registers itself using `uvm_component_utils or `uvm_object_utils. This happens automatically at compile time via a static initialiser inside the macro expansion.
2
Creation via factory — components and objects are created with ::type_id::create() instead of new(). Every such call is a factory look-up that can be redirected.
3
Overrides — before the environment is built, a test can instruct the factory to return type B whenever type A is requested. No source edits, no recompile of the original files.

What the factory is NOT

Not a run-time compiler. Not a dynamic loader. Not a simulator plugin. It is purely a SystemVerilog class-level look-up that swaps which new() is called inside type_id::create(). All classes must still be compiled before simulation starts.

Key rule from Cummings 2012 "Technically only the test must be registered with the factory. No other testbench component is required — but they should be!" Every component you skip registering loses the ability to be overridden.

The central problem it solves

In any real project you will eventually need a second transaction type with extra fields, or a second driver for serial vs parallel output. Without a factory, each change cascades through the whole hierarchy:

❌ Without factory — test2 needs:
trans2 → new trans_sequence2 driver2 → new agent2 agent2 → new env2 env2 → new test2 All files maintained separately
✅ With factory — test2 needs:
Write trans2, driver2 only One line in test build_phase: set_type_override_by_type(...) All existing files reused unchanged Regression continues to work

Concrete examples (Cummings SNUG 2012)

Example 1 — swap transaction type globally

trans2 extends trans with an extra valid bit. One factory override in test2 makes every trans::type_id::create() call return a trans2 object. The sequencer, driver, agent, and env source files are never touched.

Example 2 — swap one driver instance

tb_driver2 is a serial-output variant. An instance override at path e.agnt.drv replaces just that one driver. All other drivers in the testbench are unaffected. The agent source is never touched.

PISO application — swap to directed sequence

In your PISO testbench, piso_directed_seq already exists as a subclass of piso_seq. One type override in a directed test makes the agent run corner-case patterns without touching the env, agent, or driver.

① The custom_phase Class — Phase Descriptor

This class is the heart of the mechanism. It extends uvm_task_phase which tells UVM: "this phase requires simulated time and must be implemented as a SystemVerilog task, not a function."

custom_phase.svSystemVerilog

① local static m_inst

The single shared object. local prevents external code from touching it. static means it belongs to the class, not any instance, and persists for the entire simulation lifetime.

② local function new()

Making the constructor local is the key enforcement mechanism. No code outside this class can call new custom_phase(), so accidental duplicate phase nodes are impossible.

③ static function get()

The public entry point. On the first call m_inst == null so it creates the one object. On every subsequent call it returns the same handle. This is exactly how uvm_connect_phase::get() works internally.

④ type_name / get_type_name()

Required by UVM's phase engine for logging, debug printouts, and dm.find() lookups. Every built-in UVM phase follows this exact same pattern.

⑤ exec_task()

UVM calls this once per component in the testbench hierarchy. The $cast filters: only components that are base_comp (or its subclasses) get their custom_phase() invoked. All other components are silently skipped.

A Virtual task as interface

base_comp declares the task as virtual so subclasses can override it. The empty body is intentional — components that don't need the phase simply do nothing.

B Opt-in by inheritance

agent and env extend uvm_agent/uvm_env directly — not base_comp. So the $cast inside exec_task fails for them and they are silently excluded. Only driver and monitor (which extend base_comp) participate.

C Objection pattern

Each participating component must raise_objection before work and drop_objection after. The custom phase (being a uvm_task_phase) ends only when all objections across all components are dropped — same mechanism as run_phase.

④ How exec_task() Dispatches to Components

When UVM enters the custom phase it traverses the entire component tree and calls exec_task(comp, phase) for every component. Here is the decision chain for each call:

1
UVM phase engine calls exec_task(comp, phase) with the next component in the hierarchy traversal.
2
$cast(bc, comp) attempts to assign comp into a base_comp handle. If comp is a driver or monitor (both extend base_comp), the cast succeeds and returns true.
3
If cast succeeds → bc.custom_phase(phase) is called. Due to virtual dispatch, the overridden version in driver or monitor executes. For agent, env, test_c the cast fails (they don't extend base_comp), so the if body is skipped entirely.
4
Both driver and monitor raise an objection at entry and drop it at exit. The phase engine waits until all objections across all spawned tasks are dropped, then proceeds to the next phase.

Returns the global domain shared by all standard UVM components. Inserting into this domain ensures every component in the simulation — not just those in a custom domain — sees the new phase.

② dm.find(uvm_connect_phase::get())

Searches the domain's schedule graph for the node whose handle identity matches the singleton returned by uvm_connect_phase::get(). Returns the schedule node (a uvm_phase wrapper) which will serve as the insertion anchor.

③ dm.add() — the four arguments

arg1 custom_phase::get() — the phase to insert (singleton).
arg2 null — no parallel synchronisation.
arg3 ph — insert after connect_phase.
arg4 null — no forced upper boundary.

④ Why in the constructor?

The phase schedule must be modified before build_phase starts. Calling add_my_phase() from the constructor guarantees this timing. Doing it in build_phase would be too late — the domain schedule has already been locked by the time children are constructed.

🏭 Factory Inside run_test()

When a test name is resolved, uvm_factory.create_component_by_name() is called. Every class registered with `uvm_component_utils is in the factory lookup table at this point. If a type override was registered, the factory silently substitutes the override — no source changes needed.

📅 Phase DAG at this point

get_common_domain() builds the full phase schedule as a directed acyclic graph: build → connect → eoe → sos → run → extract → check → report → final. Run sub-phases (reset, configure, main…) are also scheduled inside the run node. This DAG drives all subsequent phase execution.

⚡ run_phase Parallel Threads

Every component's run_phase task is forked into its own thread simultaneously. The test calls phase.raise_objection(this), starts sequences, then phase.drop_objection(this). When the last objection drops across all components, the phase ends and UVM advances to extract_phase.

🔇 The Silent No-op Path

If no +UVM_TESTNAME is found, no test component is created. UVM still runs all phases with only uvm_root as a participant. Since no test raises an objection in run_phase, it ends immediately. The simulation completes in near-zero time — the classic symptom of a forgotten +UVM_TESTNAME plusarg.

📋 UVM RAL Model

The Register Abstraction Layer (RAL) is a high-level, object-oriented abstraction for accessing and verifying design registers and memories. RAL mimics the design register space, keeps predicted values in sync, and provides a bus-agnostic API that works at both block and system level — making register verification automated, scalable, and reusable.

Why RAL?

Register verification without RAL requires manually tracking every register's attributes, reset values, and field coverage ranges — error-prone and slow. RAL automates this with a structured model that mirrors the design, ready-made APIs, and built-in self-checking.

RAL Hierarchy

Three levels mirror the design register structure:

uvm_reg_block TOPMOST
Top of the RAL hierarchy. Contains all register (uvm_reg) instances and/or nested sub-blocks. Also holds the register map(s).
uvm_reg
Mimics a single hardware register. Tracks the mirror/predicted value and current (actual) value. Contains one or more uvm_reg_field instances.
uvm_reg_field
Represents a named bit-field within a register. Has its own access policy (RW, RO, W1C…), reset value, and coverage.

The Adapter Class

The adapter bridges the bus protocol and the RAL model. It must extend uvm_reg_adapter and implement two functions:

uvm_sequence_item
(bus transaction)
⇄
uvm_reg_bus_op
(RAL operation)
reg2bus()
Called for every frontdoor write/read. Translates the RAL uvm_reg_bus_op into a bus-protocol sequence item (e.g. APB or AXI).
bus2reg()
Called after the bus transaction completes. Converts the completed bus item back into a uvm_reg_bus_op so RAL can update its mirror.

uvm_mem mirrors design memory inside the RAL model. Like uvm_reg it supports frontdoor and backdoor access, and adds burst read/write capability.

What is the Predictor?

Every RAL register has a mirror value (what RAL thinks the hardware holds) and a desired value. The predictor keeps these in sync by calling predict() on a register whenever an access occurs. Three prediction modes exist:

Implicit Explicit Passive
① Implicit Prediction

Simplest approach. The RAL base class automatically calls predict() after every write()/read() initiated via the RAL API. Enable with:

env.svSystemVerilog
function void connect_phase(uvm_phase phase); regmodel.default_map.set_auto_predict(1); // disabled by default endfunction
Figure 2 — Implicit prediction flow
When to use Best for simple testbenches where all register accesses go through the RAL API. No extra infrastructure needed.
② Explicit Prediction

A uvm_reg_predictor component is connected to the monitor. The monitor captures bus transactions; the predictor converts them (via the adapter) into register operations and calls predict(). This keeps the mirror accurate even for accesses not initiated by the test.

③ Passive Prediction

Useful when register accesses are not initiated via the RAL model at all (e.g. a CPU subsystem in a full-chip testbench). The predictor observes all bus traffic passively and updates the mirror based on observed transactions. Structurally very similar to explicit prediction — same three-step setup — but set_auto_predict is left disabled.

Key difference vs Explicit Explicit prediction assumes RAL is the initiator of some accesses; Passive prediction is appropriate when RAL is purely an observer — it never drives any bus transaction.
RAL Coverage Hooks

UVM RAL provides a sample() virtual method in both uvm_reg_block and uvm_reg. Override it to trigger your covergroup whenever a register is accessed. Register prediction must be enabled for coverage to fire correctly.

Pre-defined Register Sequences

UVM ships a library of ready-made register test sequences. Point any of them at your RAL block and they exercise the entire register/memory space automatically. Individual registers or memories can be excluded by setting the NO_REG_TEST or NO_MEM_TEST attribute on that register/memory instance.

SequenceWhat it checks
uvm_reg_hw_reset_seq Reads every register and verifies the value matches the declared reset value.
uvm_reg_bit_bash_seq Walks 1s and 0s through every bit of every register; checks the result based on the field's access policy.
uvm_reg_access_seq Frontdoor write → backdoor read check; then backdoor write → frontdoor read check. Validates both paths agree.
uvm_mem_walk_seq Writes a walking-1s/0s pattern to each memory location and compares with the readback.
uvm_mem_access_seq Same frontdoor↔backdoor cross-check as uvm_reg_access_seq, applied to every memory location.
uvm_reg_shared_access_seq Writes each register via every available address map; reads back via all maps to confirm consistency.
uvm_mem_shared_access_seq Same multi-map cross-check for every memory location.
uvm_reg_mem_shared_access_seq Runs uvm_reg_shared_access_seq followed by uvm_mem_shared_access_seq.
uvm_reg_mem_built_in_seq Umbrella sequence — runs a configurable selection of all the above sequences in one shot.
uvm_reg_mem_hdl_paths_seq Verifies that every declared backdoor HDL path resolves to a valid RTL node.
